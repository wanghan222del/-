# rm_can_bridge USB-CAN 双格式接收设计

**目标**

让 `rm_can_bridge` 在保持现有 `0x301` 发送格式不变的前提下，同时兼容两种 USB-CAN 串口回传格式：
- 旧格式：`30` 字节，`55 AA ... 88`
- 新格式：`16` 字节，`AA ... 55`

**问题背景**

当前节点接收端只支持旧 `30` 字节格式，因此虽然新适配器已经把 `0x402` 的原始串口数据送到了 `/dev/ttyACM0`，但节点仍会因为帧头/帧长不匹配而直接丢弃数据，导致 `/gimbal/feedback/rpy_deg` 不发布。

**方案选择**

- 方案 A（采用）：抽出一个独立的 USB-CAN 帧解析器，同时支持旧 `30` 字节与新 `16` 字节格式；节点循环从串口缓冲中调用解析器，解析成功后统一得到 `can_id + 8字节payload`。
- 方案 B：直接在节点里堆条件分支解析两种格式。改动更快，但后续不好测，不利于继续兼容更多适配器。

选择方案 A，因为它改动小、单元测试直接、不会影响现有 `0x301` 发送逻辑。

**数据流**

1. 串口读到原始字节后追加到 `serial_rx_buffer_`
2. 解析器尝试识别一帧：
   - 若是完整旧格式帧，取 `frame[13..16]` 为 CAN ID，`frame[21..28]` 为 payload
   - 若是完整新格式帧，取 `frame[3..6]` 为 CAN ID，`frame[7..14]` 为 payload
3. 若 CAN ID 为 `0x402`，沿用现有 `publishFeedback()` 发布反馈角
4. `0x301` 发送逻辑完全保持原样

**错误处理**

- 头字节不匹配：丢弃 `1` 字节继续找同步
- 数据不足一整帧：保留缓冲区，等待下一次串口读入
- 识别到完整帧但 ID 不是 `0x402`：消费该帧但不发布反馈

**测试策略**

- 为旧 `30` 字节格式保留通过用例
- 为新 `16` 字节格式新增通过用例
- 为未知前导字节与不完整帧保留失败/等待用例
- 最后跑 `rm_can_bridge` 定向测试验证
